C++++ - A Specification

This is the official (but not formal) specification for the C++++ programming language! Although it's intended to be used by people wishing to implement a C++++ compiler, it can also be used by developers wishing to write C++++ programs. As a result, this document will not assume which position that you, the reader, takes. Instead, I'll refer to C++++ developers and compiler vendors as separate entities in this document.

A pattern that you might encounter within this specification is the lack of formal language. We still don't know how to write formal documents yet so you'll just have to deal with it. But we will try to make this document as unambiguous as possible to avoid language confusion and compiler-specific functionalities.

Background

Basically, C++++ is this system programming language that I created since I was bored. It started in late 2022 and, as of right now, currently does not possess a reference implementation. The language is currently merely a specification that anyone can implement if they wanted to.

Now, C++++ is a compiled language and should not be used as an interpreted language, though nothing is gonna prevent you from implementing a C++++ interpreter so if you wanted to, go ahead. However, as you will see layer, implementing C++++ as an interpreted language might be more difficult than a language designed for use as an interpreter, like Python.

Another thing to note about C++++ is that, despite the name, it is not backwards compatible with neither C nor C++. We have chosen to remove backwards-compatibility with C/C++ because if we're just gonna create a superset of C++, we might as well just propose a change to the C++ Standards Commitee.

Language Conventions

Naming Conventions

There are certain naming conventions used within the C++++ standard library and the rest of the language. These naming conventions are generally followed with a few exceptions (like, for example, the integer naming conventions).

  • Everything is named with snake casing (i.e. get_internal_states).
  • All the letters in the name of a constant are uppercase (i.e. MAXIMUM_PORT_NUMBER).
  • All types are postfixed with _t (i.e. dynamic_array_t)
  • The usage of abbreviations are kept to a minimum.

Please note that although the language and the standard library almost always abide to these naming conventions, developers are not required or encouraged to use them. However, to maintain consistency within the language, compiler vendors implementing extensions are recommended to follow these conventions whenever possible.

Extensions

Extensions are a way for the implementors of C++++ compilers to insert their own features into the language and developers to use those extensions easily. Other multi-implementation languages, like C++, does have extensions as well, but since the language doesn't have a concept of an extension, the difficulty of checking for the existence of an extension often outweighs the benifits the extensions provide. C++++, however, have standard support for extensions, allowing developers to easily take advantage of them.

Although compiler vendors can name their extensions however they'd like, it is strongly recommended that extension names uses snake casing (for consistency with the standard extensions and the rest of the language) and are prefixed with the compiler vendors's names (to avoid name collisions).

There are also standard extensions, basically extensions that are part of the standard C++++ specification but are not neccessarily required in an implementation of C++++. Any compiler vendor can implement any standard extensions, and they choose not to do so as well. Standard extensions allows us, the language designers, to add features that may not be applicable in every usage setting of the C++++ programming language.

Using extensions

To use any extensions, standard or not, the extension in question must first be manually enabled by the developer. This is to prevent developers from accidentally using a feature that is part of an extension thinking that it is part of the core language.

The first method for enabling extensions is by using the extension statement in the C++++ source file. This will enable the extension for the entire file, but only within the file in which the statement resides within. The syntax of this method is shown below.

extension extension_name;

Compiler vendors can also choose to add compiler flags for enabling extensions.

Fundamental Language Features

This section will describe some of the most common language features in the world of programming, like variables, functions, etc. It is going to be mainly a description of how C++++ implements a lot of these features rather than an explanation of the concepts themselves.

Data Types

Because C++++ is a statically typed language, there are often cases in which the developer needs to specify data types. C++++ has a number of data types built in to the language itself.

Integer Types

Integers are numeric values that does not contain fractional components. C++++ offers both signed and unsigned integers.

Below is a list of all the integer types offered by C++++.

"In-Code" Name Description
int8_t 8-bit signed integer
int16_t 16-bit signed integer
int32_t 32-bit signed integer
int64_t 64-bit signed integer
uint8_t 8-bit unsigned integer
uint16_t 16-bit unsigned integer
uint32_t 32-bit unsigned integer
uint64_t 64-bit unsigned integer

Functions

Functions, sometimes known as procedures in some languages, are collections of code that can be executed. The syntax for a function is shown below.

function name(parameter_name: type, ...) -> return_type_t { // Function body // ... }

To summarize:

  • function is a built-in keyword of the C++++. It is used to declare a function.
  • name is the name of the function. This name may not reflect the actual name of the function in the compiled binary, as compilers are free to mangle function names for various purposes.
  • parameter_name is the name of one of the parameters of the function, which are data passed to the function when it's called.
  • The type is the data type of parameter_name. C++++ is a statically typed language, and as a result the types of all the function parameters must be known at compile time.
  • return_type_t is the data type of the return value of the function. Omit the return type to indicate that the function does not return a value.
